# ApplyFirst Seed Data Review Checklist

Use this before inviting beta testers. The first beta does not need a perfect dataset, but the top examples must be credible enough that students trust the product direction.

## Review Goal

Manually review the highest-visibility programs so testers can judge the product workflow instead of getting distracted by stale data.

Start with:

- Programs shown as Recommended.
- Programs that appear in saved-program examples or screenshots.
- Programs that are likely to matter to freshmen and sophomores.
- Programs that appear across multiple inspiration repos.
- Programs with unclear status, old URLs, or weak official-source notes.

## Required Checks Per Program

For each priority record, confirm:

- Official URL loads.
- Program still exists.
- Program fits ApplyFirst scope.
- Eligibility/class-year fit is accurate.
- Role track is accurate.
- Opening window or expected timing is reasonable.
- Deadline is current or clearly marked as not posted.
- Funding, location, and participation format are not misleading.
- Confidence level matches the evidence.
- Status label does not overpromise.
- Source note explains what was verified.

## Beta-Ready Rule

A program is beta-ready when:

- It has an official source.
- The user-facing description is not misleading.
- The timing/status language is honest.
- The record can be saved and compared even if live alerts are not ready.

A program is not alert-ready until:

- Current-cycle timing is confirmed.
- A human reviewer has checked the official source.
- The app can avoid sending false positives.

## What To Do With Problems

- Broken URL: update the official URL or mark Needs Verification.
- Old cycle language: keep the program, but make timing conservative.
- Unclear eligibility: lower confidence and add a source note.
- Program no longer exists: remove or hide before student testing.
- Great program from a student: add as a review candidate before making it public.
